# Representing Measures in Dimensional Models

In dimensional modeling for data warehousing, **measures** are the numeric, quantitative values that you want to analyze. They are the "what" you are measuring about your business. Think of them as the data points you'd typically aggregate (sum, average, count) in a report.

## What are Measures?

Measures are typically:

*   **Numeric:** They are numbers, not text or dates.
*   **Additive, Semi-Additive, or Non-Additive:** This describes how you can combine them across dimensions.
    *   **Additive:** Can be summed across any dimension (e.g., `Sales Amount`, `Quantity Sold`).
    *   **Semi-Additive:** Can be summed across some dimensions but not others (e.g., `Account Balance` can be summed by customer but not by time).
    *   **Non-Additive:** Cannot be summed meaningfully across any dimension (e.g., `Exchange Rate`, `Ratio`). These often require special handling or are stored as is without aggregation.
*   **Facts:** Measures are stored in fact tables.

## Practical Example: Online Retail Sales

Imagine an online retail business. We want to analyze sales performance.

**Scenario:** A customer places an order. This transaction involves several numeric values we'd want to track.

Here are some potential measures:

*   `OrderQuantity`: The number of units of a specific product in an order line item. (Additive)
*   `LineItemAmount`: The total monetary value for that specific product line item (e.g., `OrderQuantity` \* `UnitPrice`). (Additive)
*   `TaxAmount`: The tax charged for that line item. (Additive)
*   `ShippingCost`: The cost of shipping for that line item. (Additive)
*   `DiscountAmount`: The discount applied to that line item. (Additive)
*   `UnitPrice`: The price of a single unit of the product. (Non-Additive if averaged across many sales, but often stored as is).
*   `CustomerAccountBalance`: If we were tracking account balances, this would be semi-additive (sum by customer, not by day).

When designing a fact table, these measures would typically form the core columns.

### Fact Table Structure (Conceptual)

| Column Name        | Data Type | Description                               | Aggregation Type |
| :----------------- | :-------- | :---------------------------------------- | :--------------- |
| `FactSalesKey`     | INT       | Surrogate key for the fact record         | N/A              |
| `DateKey`          | INT       | Foreign key to the Date dimension         | N/A              |
| `ProductKey`       | INT       | Foreign key to the Product dimension      | N/A              |
| `CustomerKey`      | INT       | Foreign key to the Customer dimension     | N/A              |
| `OrderQuantity`    | INT       | Number of units of a product sold         | SUM              |
| `LineItemAmount`   | DECIMAL   | Total price for the line item             | SUM              |
| `TaxAmount`        | DECIMAL   | Tax applied to the line item              | SUM              |
| `ShippingCost`     | DECIMAL   | Shipping cost for the line item           | SUM              |
| `DiscountAmount`   | DECIMAL   | Discount applied to the line item         | SUM              |

In this example, `OrderQuantity`, `LineItemAmount`, `TaxAmount`, `ShippingCost`, and `DiscountAmount` are all additive measures. When you query this fact table, you would sum these columns, for example, to get the total sales amount for a specific month or for a particular customer.
