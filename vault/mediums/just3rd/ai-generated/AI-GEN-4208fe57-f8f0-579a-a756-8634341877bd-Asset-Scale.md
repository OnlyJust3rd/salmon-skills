# Asset Scale: Maintaining Consistent Size in 3D Workflows

When bringing 3D assets into a real-time environment, one of the most fundamental aspects to manage is **asset scale**. Inconsistent scaling can lead to a multitude of problems, from assets appearing too large or too small relative to the scene and each other, to issues with physics, animation, and even performance. This lesson focuses on understanding and applying correct scaling practices for imported 3D assets.

## Why Consistent Scale Matters

Imagine importing a character model that's designed to be 6 feet tall, but your game world's units are set to centimeters. Without proper scaling, that character might appear as a tiny speck or a colossal giant. This is more than just an aesthetic issue.

*   **World Cohesion:** All assets should feel like they belong in the same world. Consistent scale ensures that objects have believable relationships to each other.
*   **Physics:** Colliders and physics simulations rely on accurate scale. An object that's disproportionately large or small will interact unnaturally with other physics objects.
*   **Animation:** Character and object animations are often created with a specific scale in mind. Incorrect scaling can distort animations or make them appear to move at incorrect speeds.
*   **Performance:** While not directly about polygons, extremely large or small objects can sometimes interact with rendering or culling systems in unexpected ways, though this is less common than the other issues.
*   **User Interface (UI) and Camera:** How objects appear on screen is directly tied to their scale relative to the camera and viewport.

## Understanding Real-World Units vs. Engine Units

The core of managing asset scale lies in understanding the difference between the units used in your 3D modeling software (like Blender, Maya, 3ds Max) and the units used in your target real-time engine (like Unity, Unreal Engine, Godot).

Many 3D modeling programs default to generic units. You might model something at "1 unit" length, but that "1 unit" could represent a meter, a centimeter, an inch, or even an abstract value. Real-time engines, however, often have a defined unit system. For example:

*   **Unity:** Defaults to 1 unit = 1 meter.
*   **Unreal Engine:** Defaults to 1 unit = 1 centimeter.

This mismatch is the primary cause of scaling problems.

## Practical Application: Importing and Scaling

The process of ensuring correct asset scale typically involves these steps:

1.  **Know Your Target Engine's Units:** Before you even start modeling or importing, understand the default unit system of your destination real-time engine. This is crucial for setting up your modeling software correctly.
2.  **Configure Your 3D Modeling Software:** Most 3D modeling software allows you to set your scene's unit system. Match this to your target engine's units.
    *   **Example (Blender):** In the Properties panel, under "Scene Properties," you can change the "Unit System" and "Length" units. If your engine uses meters, set Blender to Metric and Length to Meters.
3.  **Model to Scale:** When creating or preparing assets, model them to their real-world dimensions based on your chosen unit system. A door should be approximately 2 meters tall, a chair about 0.5 meters wide, a human character around 1.7-1.8 meters tall.
4.  **Check Scale on Import:** When you import an asset into your engine, pay attention to the import settings. Many engines have an "Import Scale" or "Scale Factor" option.
    *   If your model was created in meters in Blender, and your engine uses meters, the scale factor should typically be `1.0`.
    *   If your model was created in meters in Blender, and your engine uses centimeters (like Unreal Engine), you'll need to scale your imported model. A 1:1 import would result in a 1-meter object appearing as 1 cm. You'd likely need to set an "Import Scale" of `100` (1 meter = 100 centimeters).
5.  **Verify Scale in Engine:** Once imported, place your asset in a scene with other objects of known scale (e.g., a default character controller, a primitive cube of a known size) to visually confirm it looks correct.

### Scenario: Importing a Table Model into Unity

Let's say you've modeled a table in Blender, ensuring it's 1.2 meters long, 0.7 meters wide, and 0.8 meters high. You want to import this into Unity, which uses meters as its default unit.

**Steps:**

1.  **Blender Setup:** Ensure your Blender scene units are set to Metric, with Length set to Meters.
2.  **Modeling:** You've modeled your table to be precisely `1.2` units long, `0.7` units wide, and `0.8` units high, where each unit represents a meter.
3.  **Exporting:** Export your table from Blender (e.g., as an FBX file).
4.  **Unity Import:**
    *   Drag the FBX file into your Unity Project window.
    *   Select the imported FBX file in the Project window to view its import settings in the Inspector.
    *   Under the "Model" tab, check the "Scale Factor." It should be `1.0`. If it's not, change it to `1.0`.
    *   Click "Apply."
5.  **Placement:** Drag the table asset from your Project window into your Unity Scene.
6.  **Verification:** Place a Unity default cube (which is 1x1x1 meter) next to your table. The table should appear approximately 1.2 meters long, 0.7 meters wide, and 0.8 meters high, with the cube fitting proportionally.

### Scenario: Importing a Chair Model into Unreal Engine

You've modeled a chair in 3ds Max, aiming for 0.5 meters wide, 0.5 meters deep, and 0.9 meters tall. You want to import this into Unreal Engine, which uses centimeters as its default unit.

**Steps:**

1.  **3ds Max Setup:** Set your 3ds Max units to Metric, with the "Display Unit Scale" set to Centimeters.
2.  **Modeling:** Model your chair to be `50` units wide, `50` units deep, and `90` units high, where each unit is a centimeter.
3.  **Exporting:** Export your chair from 3ds Max (e.g., as an FBX file).
4.  **Unreal Engine Import:**
    *   In Unreal Engine, navigate to your Content Browser and click "Import."
    *   Select your FBX file.
    *   In the FBX Import Options dialog, find the "Scale" setting. This is often `1.0` by default, meaning 1 3ds Max unit = 1 Unreal Engine unit. Since you modeled in centimeters, and Unreal uses centimeters, this might seem correct.
    *   **Crucially, Unreal Engine's default FBX importer often assumes a generic unit unless specific conversion settings are met.** A common issue is that `1.0` scale might result in a tiny object if 3ds Max's internal units weren't perfectly aligned with the export scale.
    *   **A safer approach for this specific mismatch (3ds Max cm to UE cm) is often to ensure your 3ds Max export scale is set correctly, or to use a Scale Factor of `1.0` in Unreal and then manually scale the imported mesh within Unreal if it's not 1:1. However, if you ensure 3ds Max is set to Centimeters and export, Unreal's default `1.0` scale should interpret your 50cm model as 50cm.**
    *   If your imported chair appears too small (e.g., 0.5 units instead of 50), you would adjust the "Scale" in the import options to `100` (or manually scale the mesh in Unreal by 100 after import).
5.  **Placement:** Drag the chair asset into your Unreal Engine level.
6.  **Verification:** Place a default Unreal Engine cube (which is 100x100x100 cm or 1x1x1 meter) next to your chair. The chair should appear 0.5 meters wide, 0.5 meters deep, and 0.9 meters tall, fitting proportionally with the cube.

## Common Pitfalls

*   **Forgetting to Set Engine Units:** Assuming your modeling software's default units match your engine's is a common mistake.
*   **Scaling in Hierarchy vs. Object:** Applying scale changes at the object level is generally preferred over scaling it up in the parent hierarchy, as the latter can have unintended consequences on child objects or animations.
*   **"Flipping" Scale:** Sometimes, due to incorrect import settings or workflow, an object might be imported with a negative scale. This will cause rendering and physics issues. Always check for non-uniform scale in your engine's inspector.
*   **Ignoring Measurement:** Relying solely on visual estimation without referencing known-scale primitives or checking numerical values can lead to subtle errors.
