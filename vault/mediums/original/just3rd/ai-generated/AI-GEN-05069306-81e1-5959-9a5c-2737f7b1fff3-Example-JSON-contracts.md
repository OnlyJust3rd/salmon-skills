# Drafting Example JSON Contracts for REST Operations

When designing a RESTful API, defining the expected data structures for requests and responses is crucial. These structures, often represented in JSON, act as the "contract" between the API provider and its consumers. This lesson focuses on how to draft these example JSON contracts for core operations.

## Why Define JSON Contracts?

Clear JSON contracts achieve several important goals:

*   **Communication:** They clearly communicate the expected input and output of API endpoints to developers who will consume your API.
*   **Consistency:** They ensure that data is formatted consistently across different requests and responses, reducing ambiguity.
*   **Validation:** They provide a blueprint for validating incoming requests and outgoing responses, helping to catch errors early.
*   **Documentation:** They serve as a fundamental part of API documentation, making it easier for developers to understand and integrate with your API.

## Core Operations and Their JSON Contracts

We'll consider a simple problem domain: managing a collection of "books" in a library. For each core REST operation (GET, POST, PUT, DELETE), we'll draft example JSON contracts for both requests and responses.

### 1. Retrieving a Resource (GET)

When a client requests a specific book (e.g., `GET /books/123`), the API should return the details of that book.

**Request Example:**

For a `GET` request to retrieve a single resource, there is typically no request body. The identifier is usually part of the URL.

**Response Example (Success - HTTP 200 OK):**

This JSON represents a single book's details.

```json
{
  "id": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
  "title": "The Hitchhiker's Guide to the Galaxy",
  "author": "Douglas Adams",
  "isbn": "978-0345391803",
  "publication_year": 1979,
  "genres": ["Science Fiction", "Comedy"]
}
```

**Response Example (Not Found - HTTP 404 Not Found):**

If the requested book doesn't exist.

```json
{
  "error": "Book not found",
  "message": "The book with the ID '123' could not be found."
}
```

### 2. Creating a Resource (POST)

When a client wants to add a new book to the library (e.g., `POST /books`), they send the book's details in the request body.

**Request Example:**

The request body contains all the information for the new book, excluding the ID which will be generated by the server.

```json
{
  "title": "Pride and Prejudice",
  "author": "Jane Austen",
  "isbn": "978-0141439518",
  "publication_year": 1813,
  "genres": ["Romance", "Classic"]
}
```

**Response Example (Success - HTTP 201 Created):**

Upon successful creation, the server typically responds with the newly created resource, including its assigned ID, and a `Location` header pointing to the resource's URL.

```json
{
  "id": "f0e1d2c3-b4a5-6789-0123-456789abcdef",
  "title": "Pride and Prejudice",
  "author": "Jane Austen",
  "isbn": "978-0141439518",
  "publication_year": 1813,
  "genres": ["Romance", "Classic"]
}
```

**Response Example (Bad Request - HTTP 400 Bad Request):**

If the request body is missing required fields or has invalid data.

```json
{
  "error": "Invalid input",
  "message": "The 'title' field is required."
}
```

### 3. Updating a Resource (PUT)

When a client wants to update an existing book (e.g., `PUT /books/a1b2c3d4-e5f6-7890-1234-567890abcdef`), they send the complete, updated representation of the book.

**Request Example:**

The request body should contain the full representation of the resource as it should be after the update.

```json
{
  "id": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
  "title": "The Hitchhiker's Guide to the Galaxy (Revised Edition)",
  "author": "Douglas Adams",
  "isbn": "978-0345391803",
  "publication_year": 1979,
  "genres": ["Science Fiction", "Comedy", "Satire"]
}
```

**Response Example (Success - HTTP 200 OK or HTTP 204 No Content):**

A successful update can return the updated resource or simply confirm success without returning a body.

```json
{
  "id": "a1b2c3d4-e5f6-7890-1234-567890abcdef",
  "title": "The Hitchhiker's Guide to the Galaxy (Revised Edition)",
  "author": "Douglas Adams",
  "isbn": "978-0345391803",
  "publication_year": 1979,
  "genres": ["Science Fiction", "Comedy", "Satire"]
}
```

**Response Example (Not Found - HTTP 404 Not Found):**

If the book to be updated does not exist.

```json
{
  "error": "Book not found",
  "message": "The book with the ID '123' could not be found for update."
}
```

### 4. Deleting a Resource (DELETE)

When a client wants to remove a book from the library (e.g., `DELETE /books/a1b2c3d4-e5f6-7890-1234-567890abcdef`).

**Request Example:**

Similar to `GET`, `DELETE` requests typically don't have a request body. The identifier is in the URL.

**Response Example (Success - HTTP 204 No Content):**

A successful deletion usually returns an empty response body with a 204 status code.

**Response Example (Not Found - HTTP 404 Not Found):**

If the book to be deleted does not exist.

```json
{
  "error": "Book not found",
  "message": "The book with the ID '123' could not be found for deletion."
}
```

## Key Considerations When Drafting JSON Contracts

*   **Data Types:** Be explicit about data types (e.g., string, number, boolean, array, object).
*   **Required vs. Optional Fields:** Clearly indicate which fields are mandatory.
*   **Naming Conventions:** Use consistent naming conventions (e.g., camelCase or snake_case).
*   **Error Handling:** Define a standard format for error responses.
*   **Identifiers:** Specify how unique identifiers are represented (e.g., UUIDs, integers).
*   **Arrays:** When dealing with collections, ensure the array element structure is also defined.

By carefully drafting these example JSON contracts, you lay a solid foundation for building robust and developer-friendly RESTful APIs.
