# SPI Device Interfacing with AVR Microcontrollers

This lesson focuses on the practical application of the Serial Peripheral Interface (SPI) protocol to connect and communicate with external devices using AVR microcontrollers. This is a crucial skill within the broader context of interfacing microcontrollers with peripherals.

## Understanding SPI

SPI is a synchronous serial communication protocol designed for short-distance communication, primarily used for connecting microcontrollers to peripheral devices like sensors, memory chips, and displays. It's a full-duplex protocol, meaning data can be transmitted and received simultaneously.

### Key Components of SPI

*   **Master Device:** The microcontroller that initiates and controls the communication.
*   **Slave Device:** The peripheral device that responds to the master.
*   **Master Out, Slave In (MOSI):** Data line from the master to the slave.
*   **Master In, Slave Out (MISO):** Data line from the slave to the master.
*   **Serial Clock (SCK):** Generated by the master to synchronize data transfer.
*   **Slave Select (SS) or Chip Select (CS):** An active-low signal from the master to select a specific slave device. When SS is high, the slave ignores communication.

### How SPI Works

The master device initiates communication by asserting the Slave Select (SS) line for the desired slave device. Once the slave is selected, the master generates a clock signal (SCK). For each clock pulse, one bit of data is shifted out from the master on the MOSI line and simultaneously shifted into the slave. In the opposite direction, one bit is shifted out from the slave on the MISO line and shifted into the master. This process continues for the entire data packet, typically byte by byte. The master deasserts the SS line to end the communication with that specific slave.

## Interfacing an SPI Device: A Practical Example

Let's consider interfacing an SPI-based Inertial Measurement Unit (IMU) sensor with an AVR microcontroller (e.g., an ATmega328P found on Arduino boards). This IMU will provide acceleration and gyroscope data.

### Hardware Connections

You will need to connect the following pins between the AVR microcontroller and the IMU sensor:

| AVR Microcontroller Pin | IMU Sensor Pin | Description                 |
| :---------------------- | :------------- | :-------------------------- |
| SCK                     | SCK            | Serial Clock                |
| MOSI (or COPI)          | MOSI (or COPI) | Master Out, Slave In        |
| MISO (or CIPO)          | MISO (or CIPO) | Master In, Slave Out        |
| Digital Pin (e.g., D10) | SS (or CS)     | Slave Select (chosen pin)   |
| GND                     | GND            | Ground                      |
| VCC                     | VCC            | Power (e.g., 3.3V or 5V)    |

**Important:** Always refer to the datasheets for both your AVR microcontroller and the specific SPI device for pin configurations and voltage compatibility.

### Software Implementation (Conceptual C Code for AVR-GCC)

The AVR microcontrollers have built-in SPI hardware (SPI peripheral). You'll need to configure it and then write code to send and receive data.

```c
#include <avr/io.h>
#include <util/delay.h>

// Define the Slave Select pin
#define SS_PIN PB2 // Example: Using PB2 on ATmega328P, often D10 on Arduino

void spi_init_master(void) {
    // Set MOSI, SCK, and SS as output
    DDRB |= (1 << DDB2) | (1 << DDB3) | (1 << DDB5); // SS, MOSI, SCK
    // Set MISO as input
    DDRB &= ~(1 << DDB4); // MISO

    // Enable SPI, Master mode, set clock rate fck/16
    SPCR = (1 << SPE) | (1 << MSTR) | (1 << SPR0);
}

void spi_transmit_byte(uint8_t data) {
    // Start transmission
    SPDR = data;
    // Wait for transmission complete
    while (!(SPSR & (1 << SPIF)));
}

uint8_t spi_receive_byte(void) {
    // Start reception (by sending dummy data)
    SPDR = 0xFF; // Sending 0xFF to trigger clock and receive
    // Wait for reception complete
    while (!(SPSR & (1 << SPIF)));
    return SPDR;
}

void spi_device_select(void) {
    PORTB &= ~(1 << SS_PIN); // Set SS low to select the device
}

void spi_device_deselect(void) {
    PORTB |= (1 << SS_PIN); // Set SS high to deselect the device
}

int main(void) {
    spi_init_master();

    // Example: Reading a register from the IMU sensor
    // Let's assume register address 0x75 is WHO_AM_I (Device ID)

    spi_device_select();
    spi_transmit_byte(0x75 | 0x80); // Address for reading, MSB set to 1 for read
    uint8_t device_id = spi_receive_byte();
    spi_device_deselect();

    // Now 'device_id' holds the value read from the IMU's WHO_AM_I register.
    // You can then process this value (e.g., print it via UART for debugging)

    while (1) {
        // Main loop for further communication (e.g., reading accel/gyro data)
        // You would typically repeat the select, transmit (command/address), receive, deselect sequence.
        _delay_ms(100);
    }
    return 0;
}
```

### Explanation of the Code Snippet

1.  **`spi_init_master()`**:
    *   Configures the direction of the SPI pins (MOSI, SCK, SS as outputs, MISO as input).
    *   Enables the SPI peripheral (`SPE`), sets it to Master mode (`MSTR`), and configures the clock speed. `SPR0` selects fck/16.
2.  **`spi_transmit_byte(uint8_t data)`**:
    *   Writes data to the `SPDR` register, initiating the SPI transfer.
    *   Waits until the `SPIF` flag in `SPSR` is set, indicating the transfer is complete.
3.  **`spi_receive_byte(void)`**:
    *   Writes `0xFF` to `SPDR`. This is a common technique to clock in data from the slave when you don't need to send any specific data, or as part of a read command.
    *   Waits for the transfer to complete.
    *   Returns the received byte from `SPDR`.
4.  **`spi_device_select()` and `spi_device_deselect()`**:
    *   These functions control the Slave Select (SS) pin. Asserting SS (pulling it low) tells the specific slave device to listen to the master. Deselecting it (pulling it high) makes the slave ignore the communication.
5.  **`main()` function**:
    *   Initializes the SPI master.
    *   Demonstrates a typical read operation:
        *   Selects the device.
        *   Transmits the read command (register address with the read bit set).
        *   Receives the data byte from the slave.
        *   Deselects the device.

## Common Pitfalls and Tips

*   **Incorrect Pin Connections:** Double-check your wiring against the datasheets. A single misplaced wire can prevent communication.
*   **Voltage Levels:** Ensure the voltage levels of your AVR microcontroller and the SPI device are compatible. Mismatched voltage levels can damage the components.
*   **Slave Select Logic:** Remember that SS is typically active-low. If you forget to deselect a device, it might continue to interfere with other SPI communications, even if you've selected a different slave.
*   **Data Order (MSB vs. LSB):** Most SPI devices transmit data with the Most Significant Bit (MSB) first. Ensure your code respects this.
*   **Timing:** While SPI is synchronous, some devices might have specific timing requirements. Check the slave device's datasheet for details on clock speed limits and setup/hold times.
*   **Multiple Slaves:** When interfacing multiple SPI devices, each device needs its own dedicated SS pin. The MOSI, MISO, and SCK lines are shared among all slaves.

By understanding these principles and practicing with sample code and hardware, you can confidently interface various SPI-based peripherals with your AVR microcontrollers.
