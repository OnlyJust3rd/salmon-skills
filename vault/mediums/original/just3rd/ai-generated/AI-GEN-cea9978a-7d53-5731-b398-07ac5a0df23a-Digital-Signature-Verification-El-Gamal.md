# Verifying El Gamal Digital Signatures

This lesson explains how to verify a digital signature created using the El Gamal cryptosystem, a crucial part of ensuring message authenticity and integrity.

El Gamal is an asymmetric encryption algorithm that can also be used for digital signatures. A digital signature provides assurance that the message came from the claimed sender and that it hasn't been altered. The verification process checks if the signature mathematically matches the message and the sender's public key.

## How El Gamal Signatures are Verified

When someone sends a message signed with their El Gamal private key, they also send their public key and the digital signature. To verify this signature, you, as the recipient, need:

1.  The original message \(m\).
2.  The sender's public key, which consists of three values: \(p\) (a large prime number), \(g\) (a generator modulo \(p\)), and \(h\) (where \(h = g^x \pmod{p}\), and \(x\) is the sender's private key).
3.  The digital signature, which consists of two values: \(r\) and \(s\).

The verification process involves checking if a specific mathematical relationship holds true. The core idea is to re-derive a value from the message and signature using the sender's public key, and see if it matches a value derived from the sender's public key and the message.

The verification equation for an El Gamal signature \((r, s)\) on a message \(m\) using public key \((p, g, h)\) is:

\[
g^m \equiv h^r \cdot r^s \pmod{p}
\]

If this equation holds true, the signature is considered valid.

## Practical Scenario: Verifying a Transaction

Imagine you are a participant in a decentralized system. A transaction request arrives, claiming to be from a specific user. To ensure the transaction is legitimate, you need to verify the digital signature attached to it.

Let's say:

*   The message \(m\) represents the transaction details (encoded as a number).
*   The sender's public key is \((p=23, g=5, h=17)\).
*   The received signature is \((r=10, s=15)\).

To verify:

1.  **Calculate the left side of the equation:** \(g^m \pmod{p}\).
    *   We need the message \(m\). For demonstration, let's assume \(m=7\).
    *   Left side: \(5^7 \pmod{23}\).
    *   \(5^7 = 78125\).
    *   \(78125 \div 23 = 3396\) with a remainder of \(17\).
    *   So, \(5^7 \pmod{23} = 17\).

2.  **Calculate the right side of the equation:** \(h^r \cdot r^s \pmod{p}\).
    *   Right side: \(17^{10} \cdot 10^{15} \pmod{23}\).
    *   This involves modular exponentiation.
    *   \(17^{10} \pmod{23}\) can be calculated. Let's use a calculator or modular exponentiation tool for this. \(17^{10} \equiv 1 \pmod{23}\).
    *   \(10^{15} \pmod{23}\). \(10^{15} \equiv 10 \pmod{23}\).
    *   Right side: \(1 \cdot 10 \pmod{23} = 10\).

3.  **Compare the results:**
    *   Left side = \(17\).
    *   Right side = \(10\).

Since \(17 \neq 10\), this signature would be considered invalid for this message and public key. This indicates either the message was tampered with, or the signature was not generated by the corresponding private key.
